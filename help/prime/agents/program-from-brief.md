# 從簡介建立方案

在[!DNL Adobe Journey Optimizer B2B Prime]中，[_方案_](../marketing/programs.md)&#x200B;是行銷活動中歷程、人員清單、權杖和設定的最上層容器。 從[聊天介面](./chat-interface.md)，「方案建立」技能會從您的簡報（方案本身、子資料夾、權杖、人員清單和人員歷程）中以一個引導式對話來建立端對端的整個結構。

* **技能** - `program-creation`
* **引動** — 上傳簡訊並輸入&#x200B;_從此簡訊建立方案_ / _從簡訊建立方案_，或直接說明行銷活動
* **從**&#x200B;讀取/寫入 — [!DNL Journey Optimizer B2B Prime]；也會從您的租使用者讀取程式型別組態

## 簡短上傳

按一下聊天輸入中的&#x200B;_附加_&#x200B;圖示以上傳檔案，然後說明您想要的內容。 系統會擷取簡介的文字，並作為內容傳遞給助理(您會在聊天中看到「行銷活動簡介上傳：檔案名稱(NkB)」指標)。

### 支援的檔案類型

| 格式 | 行為 |
|---|---|
| `.txt`, `.md` | 直接在瀏覽器中讀取 |
| `.pdf`, `.docx`, `.xlsx`, `.json`, `.csv` | 已傳送到後端文字擷取服務 |

名稱包含&#x200B;*brief*&#x200B;的檔案會自動標示為行銷活動簡報；否則，助理會從內容推斷檔案名稱。

### 大小限制

在到達助理之前，簡短文字會截斷至約30,000個字元（~7,500個權杖）。 超長內文在該點被切斷（以`(truncated to 29KB)`註記表示）。 將基本的行銷活動細節放在前面。

## 建議的簡短內容

流量輸入會讀取以下輸入的摘要 — 儘可能包括：

* 計畫名稱和簡短說明/目的
* 程式型別訊號（例如商展、活動、網路研討會、路演、會議或培養）
* 人員清單的對象條件
* 歷程設計 — 計數、進入條件、接觸點/時間以及上線日期
* 代號（事件日期、地點、主旨列等可重複使用的值）

AI助理會在建置之前要求任何遺漏的專案。

## 建置階段

技能會以三個階段執行： **INTRACT→核准→組建**。 在您核准之前，系統不會建立任何專案。

### 輸入

助理員會從摘要中擷取下列內容，並提示您輸入任何遺漏的專案：

* 計畫名稱
* 父資料夾（若未指定，則預設為工作區根）
* 說明
* 方案型別（可能從簡短措辭推斷）
* 要建立的權杖（名稱、型別、值）
* 人員清單對象條件
* 歷程 — 計數、登入條件、接觸點、上線日期

### 核准

助理員會在繼續之前顯示確認摘要：

`I'll create {program} in folder {id}, with {N} token(s), {N} dynamic people list(s), and {N} journey(s). Shall I proceed?`

在您以明確的繼續回應（例如&#x200B;_繼續_、_繼續_、_建置它_、_已核准_&#x200B;以及&#x200B;_是_）之前，不會呼叫任何建立工具。

### 建置

建置步驟會以固定順序執行；每個步驟都取決於上一個步驟產生的ID。

| 步驟 | 說明 | 工具 | 輸出 |
|---|---|---|---|
| **1. 資料夾** | 依名稱解析父資料夾或使用工作區根目錄；可以建立新的子資料夾 | `getRootTree`, `browseFolders`, `createFolder` | 資料夾名稱+ ID |
| **1.5. 程式型別** | 查詢租使用者程式型別並挑選相符專案以取得簡介 | `browseProgramTypes` | 輸入名稱+ `programTypeId` |
| **2. 程式** | 在已解析的資料夾下建立方案，並繫結至所選型別 | `createProgram` | 計畫名稱、ID、型別 |
| **3. Token** | 在程式中建立每個`my.*`權杖 | `createProgramToken` | 權杖名稱 |
| **4. 人員清單** | 從對象條件產生屬性型規則並建立動態人物清單 | `generate_ruleset`, `createSmartListWithRules` | 清單名稱+ `smartListId` |
| **5. 歷程** | 使用有線至步驟4清單的人員受眾進入節點，建立每個人的歷程 | `create_journey` | 歷程名稱、ID |
| **6. 發佈** | 如果已設定及確認上線日期，則發佈或排程歷程 | `publish_journey_with_dates` | 即時/排程歷程 |
| **7. QA** | 選擇性驗證預覽和完整QA檢查 | `qa_program_preview`, `qa_program` | 通過/失敗/警告報告 |

**_Step相依性:_**

* 必須先取得程式ID （步驟2），才能附加權杖、清單或歷程。
* 步驟4的`smartListId`在步驟5中傳遞到`create_journey` — 省略它會使歷程的對象專案節點留空。
* 歷程提示一律以&#x200B;_Person對象節點作為進入點_&#x200B;開始，以確保有填入的進入節點。

## 輸出資產

### 方案

頂層容器。 其詳細資料檢視會顯示下列標籤：

| 標記 | 內容 |
|---|---|
| **概觀** | 名稱、說明、方案型別、狀態、資料夾位置、時間戳記 |
| **屬性** | 型別定義的自訂欄位（只有在程式型別啟用時） |
| **代號** | 此方案範圍的`my.*`權杖 |
| **歷程** | 計畫中包含的歷程 |

### 權杖

範圍設定為程式的可重複使用`my.*`值（例如`my.eventDate`）。 每個都有一個型別 — `text`、`date`、`rich text`、`score`或`number`之一 — 和一個值。 它們解析方案範圍內的內部電子郵件和內容。

### 人員清單

動態（智慧）人物清單是根據對象條件建立，透過產生的屬性型規則，在方案下建立。

### 歷程

人員歷程從繫結至步驟4人員清單的「人員」對象專案節點開始，接著是簡介中的接觸點（例如，_在1天後傳送歡迎電子郵件，3天後跟進_）。 如果設定上線日期，則可立即發佈或排程歷程。

## 程式型別解析

程式在建立時將&#x200B;**永久繫結**&#x200B;到租使用者定義的程式型別 — 這之後無法變更。 流程會透過`browseProgramTypes`在每個案例中明確解析型別。

| 案例 | 行為 |
|---|---|
| **多個可用型別** | 比對簡短文字與型別（例如tradeshow/booth/expo = *Tradeshow*/*Event*；網路研討會= *網路研討會*/*Event*；nurture/drip = *Nurture*；無清除訊號= *預設*）。 如果找不到相符專案，助理會列出可用的型別並要求。 |
| **僅限預設的租使用者** | 使用&#x200B;*預設*，並注意管理員可以新增自訂[程式型別](../admin/program-types.md)。 |
| **未設定任何型別** | 停止 — 建立會失敗。 在重試之前提示管理員布建程式型別。 |

## 預設值

| 設定 | 預設 |
|---|---|
| **資料夾** | Workspace根目錄（若未指定資料夾） |
| **程式型別** | *預設* （若沒有簡短的訊號且沒有手動選取） |
| **Personalization屬性** | 預設包含衍生角色、衍生目的、參與階層（可選擇退出） |
| **發佈** | 未來上線日期非自動 — 以手動`ACTIVATE journey by {date}`步驟記錄；僅在現在啟動且確認時立即發佈 |
| **核准門** | 明確核准前不會建立任何資產 |

## 限制

| 限制 | 詳細資料 |
|---|---|
| **簡短長度上限** | ~30K個字元；截斷的摘要會遺失尾端內容 — 先輸入要點 |
| **不可變程式型別** | 建立後即無法變更；請先驗證正確的型別，再進行核准 |
| **必要的內部欄位** | `parent`和`programTypeId`一律已設定；省略`parent`會造成存取遭拒；省略型別無訊息地退回&#x200B;*預設* |
| **歷程需要人員清單** | 步驟4中的`smartListId`是步驟5的必要專案；流程會強制執行 |
| **非同步清單成員資格** | 靜態（從條件）清單會在數分鐘內填入 — 非即時 |
| **錯誤處理** | 報告工具失敗時會有確切的錯誤；助理會提示確認輸入並重試 |
| **名稱衝突** | 未自動解析 — 將要求不同的名稱 |
| **領域** | 僅限Journey Optimizer B2B Prime；[!DNL Marketo Engage]名使用者應使用個別的方案建立/方案規劃技能 |


## QA檢查

建置完成後，AI助理會提供：

> 「您是否希望我在啟動前執行QA檢查以驗證所有內容？」

確認執行`qa_program_preview`，接著執行`qa_program`。 此程式會傳回檢查通過、失敗和任何警告的報告，以及建議的後續步驟。
